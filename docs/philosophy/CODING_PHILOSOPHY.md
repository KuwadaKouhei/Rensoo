# 実装思想 (CODING_PHILOSOPHY)

- 対象: Rensoo
- 最終更新: 2026-06-26

## 基本スタンス: 型厳格 ＋ エラー明示

型でバグを未然に防ぎ、エラーは隠さず適切に伝搬・表示する。
ユーザーのグローバル方針（CLAUDE.md）に準拠する。

## 型・言語方針

1. **TypeScript strict** を有効にする（`strict: true`）。`any` は原則禁止、必要時は理由コメントを付ける。
2. ドメインの値・状態（ノード、エッジ、マップ、生成設定）は**型で表現**し、不正状態を作りにくくする。
3. 外部入出力（LLM応答・APIリクエスト/レスポンス・フォーム入力）は**スキーマ検証**してから型付き値として扱う。

## エラーハンドリング方針

1. **try-catch で握りつぶさない**（黙って `catch {}` しない）。握る場合は必ずログ＋ユーザー通知 or 再スロー。
2. エラーは**意味のある型/メッセージ**で表現し、発生源から呼び出し側へ適切に伝搬する。
3. **ユーザー向けメッセージは日本語**。LLM失敗・ネットワーク失敗は再試行可能なUIを出す（要件 FR-6 / AC-12）。
4. 想定外入力は早期に弾く（入力検証）。アプリ全体を落とさない（NFR-8）。

## セキュリティ実装方針

1. **LLM APIキー等のシークレットはサーバー側のみ**。フロントのソース/ネットワークに露出させない（NFR-5 / AC-13）。
2. `.env` 等の機密ファイルは読まない・コミットしない（GIT_CONVENTIONS と整合）。
3. 認可は「本人のマップのみアクセス可」をサーバー側で必ず担保する（FR-22 / AC-11）。

## コードスタイル / 可読性

1. 命名は意図が伝わる日本語ドメイン語の英訳で統一（例: `associationWord`, `mindMapNode`）。
2. 関数は小さく単一責務。副作用（API/DB呼び出し）はドメインの純粋ロジックから分離する。
3. コメントは「なぜ」を書く。自明な「何を」は書かない。周囲のコードの密度に合わせる。
4. フォーマッタ/リンタ（技術選定で確定: Prettier + ESLint 等）に従い、手動整形しない。

## 変更時の運用（ユーザー方針）

- 何かを変更したら、**最後にどこを変更したかを日本語で説明**する。
- 設計と異なる実装をする場合は手を止めて相談する（思想からの逸脱は理由を残す）。
- コードを変えたら関連ドキュメント（DESIGN等）も一緒に更新する。
