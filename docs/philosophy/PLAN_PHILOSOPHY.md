# 設計思想 (PLAN_PHILOSOPHY)

- 対象: Rensoo（キーワード→LLM連想語→マインドマップ自走展開アプリ）
- 最終更新: 2026-06-26

## 基本スタンス: 拡張性重視

将来の機能追加（共有・エクスポート・多言語・LLMプロバイダ差し替え等）を見越し、
**変わりやすい部分を抽象化し、拡張点をあらかじめ用意する**ことを最優先の価値とする。
ただし「拡張性のための過剰な抽象化」は避け、拡張点は**実際に変動が予見される箇所に限定**する。

## 重視する設計原則

1. **境界の明確化 / 関心の分離**
   - 「連想語生成（LLM）」「マインドマップの状態・描画」「永続化（保存）」「認証」を疎結合に分ける。
   - UI はドメインロジックを直接持たず、ロジックは UI 非依存にする。

2. **変動点の抽象化（拡張点）**
   - **LLMプロバイダ**: 連想語生成は「入力語→連想語リスト」という*インターフェース*の背後に隠す。
     Claude / OpenAI / Gemini を差し替えてもアプリ本体に影響しないようにする（FEASIBILITY の申し送りに準拠）。
   - **永続化**: 保存先（DB/サービス）を抽象化し、ストレージ実装の変更がドメインに波及しないようにする。
   - **連想ソース**: 将来 LLM 以外（辞書API等）の連想ソースを足せる余地を残す。

3. **依存方向は内向き**
   - ドメイン（連想・マップのモデル）は外部詳細（フレームワーク・API・DB）に依存しない。
   - 外側（UI・API・DB）が内側（ドメイン）に依存する。

4. **拡張しやすい単位＝縦スライス**
   - 機能は「キーワード入力→生成→表示」のように*縦に*1機能として成立させ、
     1機能=1タスク=1ブランチ=1PR で足せる構造にする（TASKS と整合）。

5. **コスト/暴走をアーキテクチャで制御**
   - 自走展開の停止条件（最大深さ・総ノード上限）は UI 任せにせず、生成オーケストレーション層の責務とする。

## やらないこと（過剰設計の回避）

- 現時点で変動が予見できない箇所への投機的な抽象化はしない。
- マイクロサービス化・過度なレイヤリングはしない（MVP規模に見合う構成にする）。
- 「いつか使うかも」のための汎用フレームワーク化はしない。

## 逸脱時のルール

- この思想から外れる設計判断をする場合は、`docs/DESIGN.md` に**理由を明記**する。
- 拡張点を増やす/減らす判断は「実際に変動が予見されるか」を基準にする。
