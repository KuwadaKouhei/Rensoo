# テスト思想 (TEST_PHILOSOPHY)

- 対象: Rensoo
- 最終更新: 2026-06-26

## 基本スタンス: 重要部分を絞ってテストする

網羅率を目的化せず、**壊れると痛い／ロジックが複雑な箇所**に投資する。
カバレッジ数値より「重要な受け入れ条件が自動で守られている」状態を目指す。

## テスト対象の優先度

### 重点的にテストする（必須）
1. **連想語生成ロジック**: LLM応答のパース/検証、件数設定の反映、重複・空・無関係語の除去（FR-4, FR-5, AC-2）。
   - LLM呼び出し自体は**モックして**ロジックを検証する（実APIは叩かない）。
2. **自走展開の停止条件**: 最大深さ・総ノード上限で確実に停止すること、停止操作の反映（FR-13, FR-14, AC-3, AC-6）。
   - コスト暴発に直結する最重要ロジック。境界値（上限ちょうど）を必ずテスト。
3. **自動／手動モードの分岐**: 手動時に自動連鎖しないこと（FR-10〜12, AC-4, AC-5）。
4. **ノードの手動編集**: 追加/編集/削除と、削除時にエッジが孤立しないこと（FR-15〜17, AC-7）。
5. **認証・認可と保存**: ゲストは保存不可→ログイン要求、本人のマップのみアクセス可（FR-18〜22, AC-8〜11）。
   - サーバー側の認可は重点的に。

### 軽めでよい / 手動確認中心
- マインドマップの細かな見た目・レイアウト（ズーム/パン等）。
- スタイリング、アニメーション。

## テストの種類とバランス

- **ユニットテスト**を主軸にする（生成・停止条件・編集・認可などのコアロジック）。
- **主要フローの統合テスト**を補助的に置く（キーワード→生成→表示、保存→再取得 など）。
- E2E は MVP では最小限（主要ハッピーパス1〜2本）にとどめ、過剰投資しない。
- ピラミッドの比率目安: ユニット多め > 統合 中 > E2E 少。

## モック方針

- **外部依存（LLM API・認証プロバイダ・必要なら DB）はモック/スタブ**してロジックを決定的にテストする。
- ドメインの純粋ロジックはモックなしで素直にテストできるよう設計する（CODING/PLAN 思想と整合）。

## 受け入れ条件との対応

- `docs/REQUIREMENTS.md` の AC-1〜13 のうち、上記「重点」に挙げた項目はテストで担保する。
- 各テストには対応する AC 番号をコメント等で紐づけ、トレーサビリティを持たせる。

## CIゲート方針（技術選定で具体化）

- 重点テストとリンタ/型チェックを CI で実行し、グリーンを維持する（Git運用・TASKS と整合）。
